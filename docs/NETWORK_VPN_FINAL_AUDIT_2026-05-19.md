# Финальный сводный отчет по аудиту сети/VPN (2026-05-19)

## Контекст и цель
Проведен полный аудит проблем с ошибками вида:
- `i/o timeout`
- `dns-remote failed`
- `dns-direct failed`
- `use of closed network connection`
- `UNAVAILABLE`
- `network is unreachable (IPv6)`

Цель: проверить, есть ли дефект соединения в самом приложении, есть ли регрессия, и как practically улучшить стабильность/скорость без удаления «плохих» узлов.

## Что было сделано
1. Пересобран Android APK из `B:\1CODING\zeon-app\scripts` и установлен на устройство по USB (`SM-S928B`, `adb`).
2. Выполнен запуск приложения, выдача VPN permissions, многократные циклы подключения/переподключения.
3. Сняты данные с устройства:
- UI dump/screenshot
- `ip route`
- `logcat` с фильтрацией по сетевым ошибкам
- конфиги и логи из `Android/data/com.zeon.hiddify/files`
4. Выполнены проверки RU-сервисов (`yandex`, `wildberries`, `ozon`) на активном VPN.
5. Выполнены измерения latency и стабильности в циклах reconnect.

Вывод: базовая связность клиента с сервером есть, «приложение не может подключиться вообще» не подтверждено.

### 3) Задержка/потери

## Корневой вывод по проблеме
На основании аудита:
1. Критичного дефекта сетевого подключения в самом клиенте не подтверждено.
2. Основная зона риска — качество отдельных узлов/маршрутов/транспортов и неоднородность DNS-политик между профилями.
3. Исторические `i/o timeout` и `dns-* failed` с высокой вероятностью интермиттирующие (узел/сеть/маршрут), а не постоянная поломка приложения.

## Что уже улучшено
- В проекте внедрены изменения для DNS/поведения селектора в unstable-профиле (ранее выполненные правки в core-конфиг-генерации).
- Проведена device-валидация после сборки и установки.

## Рекомендации (без удаления узлов)

### P0
1. Унифицировать DNS-политику во всех профилях: primary+secondary DoH, избегать single UDP resolver.
2. Для RU-критичных маршрутов на проблемных сетях держать `prefer_ipv4/ipv4_only`.
3. Оставить health-based адаптацию узлов (снижение веса при деградации, не blacklist навсегда).

### P1
1. Sticky + hysteresis для уменьшения churn при кратковременных провалах.
2. Разделить профильные параметры под mobile/wifi.
3. Поддерживать fallback transport-пары на один и тот же endpoint.

### P2 (при доступе к серверам)
1. Снять узловые метрики: RTT p50/p95, handshake success, timeout rate, loss/jitter.
2. Проверить server-side лимиты/ресурсы (conntrack, fd, backlog, CPU/RAM pressure).
3. Настроить узлы по метрикам, а не по субъективному «плохой/хороший».

## Ограничения аудита
- Нет SSH/root к серверной стороне, поэтому server-internal bottleneck анализ не выполнялся напрямую.
- Некоторые HTTP speed-тесты через встроенный Android `curl` давали шумные TLS/redirect артефакты; итог по доступности и стабильности подтверждался через независимые признаки (`tun0`, route, logcat, ping, UI).
